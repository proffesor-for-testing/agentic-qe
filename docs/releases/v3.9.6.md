# v3.9.6 Release Notes

**Release Date:** 2026-04-06

## TL;DR

**Native HNSW search actually works now — and it's enabled by default again.** The previous `NativeHnswBackend` wrapped `@ruvector/router 0.1.28`'s `VectorDb`, which a fresh investigation found to have **four serious bugs**: HNSW search returned essentially random results (recall@10 ≈ 0–10% on textbook fixtures), the constructor unconditionally wrote a `vectors.db` redb file to your project root, only one instance could exist per process, and the NAPI dispose path didn't release the file lock (which was the root cause of v3.9.5's futex deadlock). v3.9.6 replaces the entire native HNSW backend with `hnswlib-node` — the canonical Hnswlib C++ reference implementation by Yury Malkov, the same code Pinecone, Weaviate, Qdrant, LangChain, and ChromaDB are built on. All four bugs are gone in one swap. Code-intelligence semantic search now returns the actual nearest neighbours.

## Upgrade

```bash
# Claude Code
claude mcp add agentic-qe -- npx -y agentic-qe@3.9.6
aqe init --auto

# Global
npm install -g agentic-qe@3.9.6
aqe init --auto
```

No flags or environment variables needed. If you set `RUVECTOR_USE_NATIVE_HNSW=false` as a v3.9.5 workaround, you can remove it — the native backend now works correctly.

## Migration: stale `vectors.db`

Users upgrading from a pre-v3.9.6 version may have an orphaned `vectors.db` file in their project root (3.5–50MB). It's a leftover from the old `@ruvector/router` backend, was never reliably retrievable anyway (the broken HNSW search couldn't find what was stored), and is safe to delete:

```bash
rm vectors.db
```

`aqe init` will detect the file and warn you about it. **It does not auto-delete** — per the project's data protection rules, AQE never touches `.db` files without explicit user confirmation. No data migration is required; the unified `memory.db` is the source of truth and is unchanged.

## What was broken in `@ruvector/router 0.1.28`

A fresh investigation reproduced the issue #399 failure deterministically and drilled into the upstream library's actual behavior with four diagnostic scripts. The drill-down revealed not one bug but four:

1. **HNSW search returned essentially random results.** Recall@10 was 0% on the project's FNV-hashed sine fixture and 10% on textbook unit-Gaussian random vectors. The graph walk could not find self-vectors when querying with a stored vector. Increasing `efSearch` to ≈ N (effectively brute-force) restored correctness but defeated HNSW's purpose. Reproduced on linux-arm64; matches the linux-x64 GitHub Actions failures originally reported in the issue.
2. **The `VectorDb` constructor unconditionally wrote a `vectors.db` redb file to the current working directory.** A multi-MB file outside `.agentic-qe/`, in a non-SQLite format that no AQE migration or backup tooling knew about. Direct violation of the unified memory architecture (`one DB, one schema`).
3. **Process-wide singleton lock.** Only one `VectorDb` instance could exist per process. Subsequent constructors threw `Database error: Database already open. Cannot acquire lock.`
4. **NAPI dispose did not release the redb file lock.** NAPI garbage collection isn't synchronous, so the lock outlived `dispose()`. Root cause of v3.9.5's futex deadlock when the indexer tried to recreate after reset.

## Why hnswlib-node

`hnswlib-node@^3.0.0` wraps Yury Malkov's reference C++ Hnswlib implementation — the same code Pinecone, Weaviate, Qdrant, LangChain, and ChromaDB are built on. It is correct, fast, battle-tested, and shipped with prebuilt native binaries for every major platform. It was already a direct dependency of this project from the legacy `src/integrations/embeddings/index/HNSWIndex.ts` code path, so **no new dependency was added**.

Same fixture, same params (M=16, efC=200, efS=100), same machine:

| | Before (`@ruvector/router 0.1.28`) | After (`hnswlib-node 3.0.0`) |
|---|---|---|
| Recall@10 (1k vectors) | 10% | **100%** |
| Top-1 for self-query | wrong vector | **correct** ✓ |
| Insert (1k × 384 dim) | ~430ms | **~324ms** |
| Auto-creates files in CWD | yes (`vectors.db`) | **no** |
| Multiple instances per process | no (lock contention) | **yes** |
| Dispose releases resources | no (NAPI GC dependency) | **yes (immediate)** |
| Native filter functions | no | **yes** |
| Runtime efSearch tuning | no | **yes (`setEf`)** |
| Resize past initial capacity | no | **yes (`resizeIndex`)** |
