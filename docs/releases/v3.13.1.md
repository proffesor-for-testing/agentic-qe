# v3.13.1 Release Notes

**Release Date:** 2026-07-23

## Highlights

A correctness and honesty release. Four reported issues, three of which come down
to the same failure: a tool told you something it had not actually established.
Coverage analysis reported numbers it never measured, test generation claimed AI
enhancement it never performed, and a security advisory reached users through a
dependency they never asked for. All three are fixed, and the reporting now
distinguishes *measured*, *estimated*, and *unknown* so the difference is visible
rather than something you have to infer.

Also adds per-agent LLM routing you can configure on disk, and QE skills for the
Codex CLI.

## Fixed

### Coverage analysis no longer reports numbers it did not measure ([#569])

On a Rust crate, `coverage_analyze_sublinear` returned `78.3%` lines, `100%`
branches and `0%` functions â€” none of it measured. It flagged `#[cfg(test)]`
blocks as untested production code, cited line numbers past the end of the file,
analysed test files as coverage targets, and attached a confidence score to all
of it. The reporter's ground truth was 69.22% lines / 72.84% functions with a
20%â€“96% per-file spread; the tool showed a flat 74â€“84% band.

Now:

- Rust crates are measured with `cargo llvm-cov` when it is available.
- Test code â€” `tests/`, `benches/`, and brace-matched `#[cfg(test)]` blocks â€” is
  excluded from production coverage.
- Line numbers can never exceed the file length.
- A metric that was not collected reports `null`, not `0` and not `100`.
- When nothing could be measured, the result says so (`estimated: true`,
  `measured: false`) instead of presenting a guess as data.
- Gaps derived from an estimate carry low confidence and an `estimated` flag, so
  they read as hypotheses rather than findings.

### `test_generate_enhanced` actually calls your LLM now ([#567])

The tool advertised AI-powered generation, but the LLM router never reached the
code path it runs â€” so it always returned the same generic scaffold regardless of
how you configured your provider. The only symptoms were a suspiciously constant
coverage estimate and a sub-20ms response.

It now routes through your configured model. The result reports `llmEnhanced`, so
a silent fall back to templates is visible. A configured-but-broken provider (bad
key, timeout, empty response) reports the fallback honestly instead of claiming
enhancement it did not perform. The `aiEnhancement: false` opt-out is honored.

The CLI was never affected â€” it reaches the generator through a different path
that was already wired correctly.

### A HIGH-severity advisory no longer reaches consumers ([#565])

`@huggingface/transformers` was declared an *optional* peer dependency. npm
installs optional peers â€” `optional` only suppresses the error when the peer
cannot be resolved â€” so every install pulled in `onnxruntime-node â†’ adm-zip` and
its advisory ([GHSA-xcpc-8h2w-3j85]). No resolvable version of that chain is
unaffected.

It is now an explicit opt-in: install it yourself only if you want local
in-process embeddings. Everyone using an embedding endpoint (the default) no
longer carries the advisory. Both load sites raise an actionable message if it is
absent.

### `codex` is selectable through the LLM router

The provider existed and worked, but was missing from the router's provider list,
so it could never actually be chosen.

### Fixes take effect immediately after an upgrade

Operation results were cached per project and keyed without the build version, so
an upgraded install could keep replaying the previous version's answers for up to
an hour â€” making a fix look like it had not worked. Set
`AQE_SESSION_CACHE_SALT` to force a cold cache at any time.

## Added

### Per-agent LLM routing you can configure on disk ([#568])

Route individual agent types through different providers via an `agentOverrides`
map in `.agentic-qe/llm-config.json` â€” a strong reasoning model for security
agents, a free local model for mechanical ones â€” without changing your global
default:

```jsonc
{
  "defaultProvider": "ollama",
  "agentOverrides": {
    "qe-security-scanner": { "provider": "claude-code", "model": "opus" },
    "qe-mutation-tester":  { "provider": "ollama" }
  }
}
```

Entries are partial â€” unset fields keep their category defaults. Changing provider
without naming a model uses that provider's default model, not the previous
provider's. Invalid entries are dropped with an explanation rather than failing
startup, and credentials are never persisted here.

See [the routing guide](../guides/llm-provider-routing.md) and
[ADR-125](../implementation/adrs/ADR-125-on-disk-agent-routing-overrides.md).

### QE skills and lifecycle hooks for the Codex CLI

`aqe init` now installs three Codex-native skills â€” `aqe-test-change`,
`aqe-review-quality`, `aqe-plan-quality` â€” under `.agents/skills/`, plus Codex
lifecycle hooks in `.codex/` and an `AGENTS.md` describing the repository's
conventions. If you drive this project with Codex rather than Claude Code, the QE
workflows are now available there natively.

### `AQE_COVERAGE_NO_EXEC=1` â€” opt out of running build tooling

Measuring coverage means running your project's tests, which executes code from
the analysed repository (`cargo` also runs `build.rs`; `npx` runs npm lifecycle
scripts). That is what you want for your own code, but is worth being able to turn
off when pointing the tool at something you do not trust. Existing coverage
reports are still read; everything else degrades to a labelled estimate.

### Security scans state how deeply they looked

Full SAST is implemented for JS/TS only; other languages get cross-language
pattern matching. Results now carry `analysisDepth`, so a zero-finding scan reads
as "nothing matched a generic pattern" rather than "no vulnerabilities".

## Changed

### BREAKING (result shape): coverage metrics are nullable

`coverage_analyze_sublinear` now returns `null` â€” not `0` â€” for any coverage
metric that was not collected. `lineCoverage`, `statementCoverage`,
`branchCoverage` and `functionCoverage` are all `number | null`.

- `0` means a **measured zero**.
- `null` means **unknown**.

Callers doing arithmetic or comparisons on these fields should handle `null`;
check the new `measured` / `estimated` flags to tell the cases apart.

This is deliberately a visible break rather than a silent wrong number. The
previous behaviour â€” coercing "we don't know" to `0` â€” is what produced the
impossible `100%` branch / `0%` function pair in #569.

LCOV artifacts are also no longer written for estimated or absent coverage: the
format has nowhere to record provenance, so emitting it would hand downstream
tools a file indistinguishable from instrumented output. The JSON artifact still
carries the full result.
