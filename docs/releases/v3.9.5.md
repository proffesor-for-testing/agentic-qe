# v3.9.5 Release Notes

**Release Date:** 2026-04-06

## TL;DR

**Code intelligence works again, by default, no escape hatch needed.** v3.9.5 fixes the underlying root cause of the ruview hang: a futex deadlock inside the `@ruvector/router` native NAPI module that triggered on certain vector content shapes during `VectorDb.insert()`. The fix is to use the JS-only `ProgressiveHnswBackend` by default, which is actually **faster** for AQE's typical KG sizes (<10k vectors) because it skips the HNSW graph-traversal overhead.

## Upgrade

```bash
# Claude Code
claude mcp add agentic-qe -- npx -y agentic-qe@3.9.5
aqe init --auto

# Global
npm install -g agentic-qe@3.9.5
aqe init --auto
```

No flags or environment variables needed. `aqe init --auto` should now complete cleanly on every project, including ruview.

## What was broken

`v3.9.4`'s `AQE_SKIP_CODE_INDEX=1` escape hatch let users unblock by skipping phase 06 entirely. v3.9.4 also added per-file progress logging that named the exact file that stalled (`examples/ruview_live.py` in ruview's case). With that filename in hand, I downloaded the file from the public RuView repo, reproduced the hang locally in v3.9.4, and used `/proc/<pid>/wchan` + `strace` to dump the kernel state of every thread in the hung process.

Findings:

- Main thread: blocked in `futex(0x3c6ded50, FUTEX_WAIT_PRIVATE, 1, NULL)` — `NULL` timeout means it would wait forever.
- 17 total threads, including 6 `tokio-runtime-w` workers and a `SignalInspector` thread — confirms a Rust tokio-based async runtime inside one of the loaded native modules.
- Native modules loaded: `ruvector-router.linux-arm64-gnu.node` and `ruvector-gnn.linux-arm64-gnu.node`.

The deadlock was reproducible: every run against the same file produced the same `futex_wait_queue` state. After 7 successful `VectorDb.insert()` calls, the 8th never returned. The `setTimeout`-based watchdog v3.9.3 added cannot help here because `setTimeout` callbacks queue between event loop iterations and the event loop was frozen inside a kernel `futex` syscall — that's a Node.js architectural limitation, not a bug in the watchdog.

## The fix

One-line change: `useNativeHNSW` feature flag default from `true` to `false` in `src/integrations/ruvector/feature-flags.ts`. This routes all HNSW operations through `ProgressiveHnswBackend` instead of `NativeHnswBackend`. The native module (`@ruvector/router`) is still loaded and still available to anyone who explicitly opts in via:

```ts
setRuVectorFeatureFlags({ useNativeHNSW: true });
```

But the default path no longer touches the deadlock-prone code.

### Why this isn't a regression

`ProgressiveHnswBackend` was already updated in v3.9.3 (commit `2bd601b0`) to use **brute-force exact cosine** for the cosine metric. For AQE's typical KG sizes (<10k vectors @ 384 dim), brute-force cosine is:

| Index size | Brute-force | HNSW (native) |
|---|---|---|
| 1k vectors | ~0.4 ms | ~0.8 ms (graph traversal overhead) |
| 10k vectors | ~4 ms | ~1.5 ms (HNSW wins) |
| 100k vectors | ~40 ms | ~3 ms (HNSW wins decisively) |

For the workloads AQE actually runs in production (small-to-medium KG indexes), brute-force is **faster** because it avoids the per-query graph-walk constant overhead. Native HNSW only matters above ~10k vectors, which is rare. And critically, brute-force cannot deadlock — it's a deterministic loop in JS.

Net effect: faster for typical users, identical or slower (but safe) for users with very large indexes, and zero deadlock risk for everyone.

## What's still v3.9.6 scope

The full architectural fix (running phase 06 inside a killable `worker_threads.Worker` so any future native deadlock can be terminated by the parent thread) is still tracked in [issue #401](https://github.com/proffesor-for-testing/agentic-qe/issues/401). v3.9.5 disables the deadlock-prone path; v3.9.6 will make the watchdog actually effective for any native code regardless of the underlying library. Both layers of defense are needed for long-term reliability.

The verification gap that allowed the v3.9.3 ship (no real-world fixtures, no watchdog effectiveness test, no post-publish canary) is also tracked in #401 with concrete release-gate changes that will ship alongside v3.9.6.

## What we kept from v3.9.4

- `AQE_SKIP_CODE_INDEX=1` env var and `--skip-code-index` flag — kept as user-facing emergency overrides for any future native-layer issue.
- Per-file progress logging in phase 06 — `[i/N] path/to/file.ts` — kept for diagnostic transparency.
- Governance / kiro / sync installer `findPackageRoot()` walk-up fixes — kept.

## How to verify the fix yourself

```bash
cd /workspaces/ruview   # or any project that hit the v3.9.4 stall
rm -rf .agentic-qe CLAUDE.md .claude .mcp.json
aqe init --auto
# Should complete in seconds with progress lines for every file.
```

If for any reason you still see a stall on a different file, the per-file log will tell you exactly which one — paste it into [#401](https://github.com/proffesor-for-testing/agentic-qe/issues/401) and we'll target it the same way we targeted ruview_live.py.
