## #565 — Dependency audit: HIGH/CRITICAL CVE reachable by consumers

Fixed in **v3.13.1**.

**Root cause.** `@huggingface/transformers` was declared an *optional* peer
dependency. That turned out to be the bug: npm installs optional peers —
`peerDependenciesMeta.optional: true` only suppresses the error when the peer
*cannot be resolved*, it does not mean "don't install". So every
`npm install agentic-qe` really did pull in
`@huggingface/transformers → onnxruntime-node → adm-zip@0.5.x`
([GHSA-xcpc-8h2w-3j85](https://github.com/advisories/GHSA-xcpc-8h2w-3j85)).

We had already made the *code* side safe — it is loaded through a lazy
`await import()` and degrades with a clear message — but making a dependency
optional at the **code** level does not make it optional at the **install** level.

**No in-range fix existed.** `adm-zip@0.6.0` is published, but the current
`onnxruntime-node@1.27.0` still declares `adm-zip: "^0.5.16"`, and
`@huggingface/transformers@4.2.0` pins `onnxruntime-node@1.24.3`. Root `overrides`
would have fixed our own audit and reached no consumer at all.

**Fix.** Removed it from `peerDependencies`. It remains a `devDependency` and is
now an explicit opt-in for anyone who wants local in-process embeddings; everyone
using an embedding endpoint (the default) no longer carries the advisory.

**Verification** — clean-room install of the published tarball:

```
added 134 packages
@huggingface/transformers absent — correct
adm-zip absent — correct
npm audit: found 0 vulnerabilities
```

The consumer-POV audit went from 4 HIGH to 0.

The generalised rule is now written down in
[ADR-115's amendment](../implementation/adrs/ADR-115-dependency-security-system.md)
and the dependency-security policy: **an optional peerDependency is
consumer-reachable and is in scope for the consumer audit.**

## #567 — `test_generate_enhanced` never invokes LLM enhancement

Fixed in **v3.13.1**. Thank you for the diagnosis — it was exactly right, down to
the file and line.

**Root cause**, confirmed as you described: the DI factory in
`src/domains/test-generation/index.ts` was registered as
`(memory) => _createTestGeneratorService(memory)`, so `llmRouter` never reached
the task-executor path. `isLLMEnhancementAvailable()` was therefore permanently
false and the ADR-051 branch unreachable under any provider configuration.

**Fix.** We took your option 1 — threaded the router through the registry →
`resolveTestGeneratorService` → `getTestGenerator` (now async, so the router is
resolved before the service is constructed, since it is captured in the
constructor).

Three things came out of it beyond the original report:

1. **The tool now tells you when it falls back.** The result carries
   `llmEnhanced`, plus `generationMode` and a `note` explaining why. Your report
   noted there was "no error or warning; the only signal is the suspiciously low,
   constant coverage estimate and sub-20 ms latency" — that signal is now explicit.
2. **A broken provider no longer claims success.** `llmEnhanced` reflects whether
   the model actually produced the code, so a 401/timeout/empty response reports
   the template fallback honestly instead of claiming enhancement.
3. **The CLI was never affected.** You flagged it as "likely affects the CLI
   (`aqe test generate`)" — we checked, and it reaches the generator through the
   domain plugin, which was already wired correctly.

**Verification** — real MCP call against your exact repro (`src/calculator.ts`):

```
llmEnhanced   : True
duration(ms)  : 23496          (was 18 ms)
mentions divide       : True
tests division by zero: True
uses toThrow          : True
```

The generated tests now reference the real signatures and assert
`toThrow('division by zero')`, instead of the generic
`should be defined` / `should handle basic operations` scaffold.

## #568 — Stable on-disk per-agent-type LLM provider routing overrides

Implemented in **v3.13.1**, following your proposed design.

Your investigation was accurate and saved us the discovery work — we took the
schema exactly as you specified it: `agentOverrides` in the existing
`.agentic-qe/llm-config.json`, keyed-object merge (like `providers`, not the
shallow-replace `fallbackChain` gets), resolution ahead of the category defaults,
reusing the `ModelPreference` field shape, with the same "never persist API keys"
discipline.

```jsonc
{
  "defaultProvider": "claude-code",
  "agentOverrides": {
    "qe-security-scanner": { "provider": "cognitum", "model": "cognitum-high" },
    "qe-mutation-tester":  { "provider": "ollama" }
  }
}
```

**Two things we found while wiring it that are worth knowing**, because they mean
the mechanism was further from working than the issue assumed:

1. **The per-agent ruleset was not merely unreachable from disk — it was
   unreachable at all.** `HybridRouter` only ever evaluates `config.rules`
   (falling back to the ~10 hardcoded `DEFAULT_QE_ROUTING_RULES`), and
   `createAllAgentRoutingRules()` was never installed into either. So overrides
   now also get *materialised* into routing rules the router actually evaluates.
2. **Routing rules match on `agentType`, and the QE domain services were not
   sending it.** All nine router-calling services now do, with a test that fails
   if a new call site omits it. Without this, a valid override parsed cleanly,
   built a rule, and still changed nothing.

Two smaller points from your report also fixed:

- **`codex` added to `ALL_PROVIDER_TYPES`.** You spotted this as "a separate,
  smaller gap" — it was slightly worse than that: `detectAvailableProvidersFromEnv()`
  iterates that list, so the provider could never be selected through the router.
- **Changing provider without naming a model** now resolves to that provider's
  own default model, not the previous provider's — otherwise
  `{ "provider": "ollama" }` would have routed to ollama asking for
  `claude-sonnet-4-6`.

Docs: [`docs/guides/llm-provider-routing.md`](../guides/llm-provider-routing.md).
Design record: [ADR-125](../implementation/adrs/ADR-125-on-disk-agent-routing-overrides.md),
which also explains why we extended `llm-config.json` rather than the BMAD-002
overlay — your reasoning on that point, which we agreed with.

On qe-court's parallel `routing` map: you are right that one routing model beats
two. Converging them is tracked as a follow-up in the ADR rather than done here,
because qe-court's vendor-diversity invariants (`≥2 distinct vendors seated`,
`jury vendor ∉ {writer, defense}`) need to survive the merge.

## #569 — `coverage_analyze_sublinear` reports fabricated data for Rust crates

Fixed in **v3.13.1**.

All five work items are addressed:

- [x] **Detect language and delegate to real instrumentation.** Rust crates are
      measured with `cargo llvm-cov` when it is installed (LCOV output, so gap
      detection gets genuine uncovered line numbers).
- [x] **Exclude `#[cfg(test)]` modules and `tests/` files.** Brace-matched, with
      Rust string literals and comments masked first so a `}` inside
      `assert_eq!("}", "}")` cannot close the block early.
- [x] **Never emit a line number beyond the file's length.** The cause was a
      `split('\n')` trailing-newline off-by-one — exactly the 448→449 you saw.
- [x] **Never report 100% branch coverage when no branch data was collected.**
      `branchCoverage` and `functionCoverage` are now `number | null`.
- [x] **Mark the result `estimated: true` when no instrumentation ran**, plus
      `measured` and `coverageMethod`.

**Verification** on a crate matching your fixture's shape:

```
branchCoverage  : null          (was 100)
functionCoverage: null          (was 0)
estimated       : True
measured        : False
coverageMethod  : static-estimation
files analysed  : ['src/jwks.rs']       (tests/verifier_matrix.rs excluded)
gap confidence  : 0.2                   (was 0.7)
riskAssessment  : unknown
```

The derived output is qualified too: gaps from an estimate carry low confidence
and an `estimated` flag, risk assessment reports `unknown`, and the branch/function
recommendations are suppressed when that data was never collected — you were right
that the "top three AI insights" were all downstream of the fabricated numbers.

**On the accompanying security-scan note:** agreed, and fixed. Full SAST is
implemented for JS/TS only; other languages get cross-language pattern matching.
Results now carry `analysisDepth`, and a zero-finding scan on a non-JS target says
explicitly that no language-specific analysis ran — so it reads as "nothing matched
a generic pattern", not "no vulnerabilities".

**One behaviour change to be aware of:** measuring coverage means running your
project's tests, which executes code from the analysed repository (`cargo` also
runs `build.rs`; `npx` runs npm lifecycle scripts). That remains the default
because it is the only way to produce a measurement, but `AQE_COVERAGE_NO_EXEC=1`
now turns it off entirely if you are pointing the tool at something you do not
trust.

**BREAKING:** coverage metrics are now `number | null` — `0` means a measured
zero, `null` means unknown. This is deliberately a visible break rather than a
silent wrong number; coercing "we don't know" to `0` is what produced the
impossible 100%/0% pair you reported.

The principle is written down as
[ADR-126](../implementation/adrs/ADR-126-measurement-provenance.md): *any tool
that emits a number a human will act on must state how that number was obtained;
if it was not measured, it must not be shaped like a measurement.*

Thank you for the report — the five independent contradictions and the
`cargo llvm-cov` ground truth made this straightforward to reproduce and to
verify.
