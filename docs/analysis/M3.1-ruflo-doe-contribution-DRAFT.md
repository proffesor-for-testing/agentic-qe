## What we were going to say (and why it's wrong)

Original draft: "ruflo's ADR-176 flywheel reports '2 compounding promotions' with a
flat human anchor — maybe the gains are noise; offer AQE's DoE/ANOVA significance
screen." **This is false.** The flywheel **already gates promotion on statistical
significance**, and does it *better* than the ANOVA we'd have proposed:

- `evolve-proof.ts:174-183`: `deltaCILow = bootstrapDeltaCILow(holdout paired deltas)`;
  `significant = deltaCILow > 0`; `promoted = accept(...) && significant`. Rule is
  versioned `accept/v1+sig` (`evolve-proof.ts:33`).
- It's a **one-sided 95% paired bootstrap CI** over per-held-out-task deltas (n≥20 on
  the generations path). For 2 groups an ANOVA F = t², **unpaired**; their **paired**
  bootstrap CI is strictly stronger. An added ANOVA-on-candidate-vs-baseline would be
  redundant AND weaker.
- The human-labeled frozen anchor is a **no-regression guard** (`redblue` term), not
  the promotion objective — so the objective is a proxy delta *guarded against*
  human-relevance regression. Also `detectPlateau` (`evolve-proof.ts:462`) already does
  variance-based "no-effect" classification.

**Correction for the strategy report's open-Q #3:** ruflo's flywheel *is* rigorously
significance-gated. Our "beads"-style concern does not apply. Credit where due.

## The only genuinely-additive idea (bigger, optional)

There is ONE thing AQE's DoE lens could add that ruflo does NOT already have — but it
targets a *different* question and is a substantial change, not a quick gate:

**A one-way ANOVA candidate-SELECTION veto.** The existing gate answers "does *this*
candidate's paired delta beat zero?" It does not answer "does *any* grid config beat
baseline beyond noise?" A one-way F-test across the k grid candidates' per-task score
vectors would veto a search round where the whole grid is within-noise of baseline —
complementary to the pairwise CI and to `detectPlateau`.

**Why it's non-trivial (the real blocker):**
1. The per-config per-task sample vectors live only transiently in the search `cache`
   (`harness-flywheel-generations.ts:211`) — **not retained in the receipt bundle**.
   The ANOVA veto needs them retained.
2. The promotion rule is **replay-verified**: `verifyReceiptBundle` (`evolve-proof.ts:291`)
   independently recomputes the decision and asserts `decisionMatches`; any new gate term
   MUST be recomputable purely from bundle-embedded data AND keyed by a **bumped
   `PROMOTION_RULE_VERSION`** (e.g. `accept/v2+anova`) that the verifier branches on — or
   every new receipt fails independent replay.
3. Opt-in via a new `RUFLO_HARNESS_ANOVA` env (mirroring `RUFLO_HARNESS_LOOP`).

So this is a real feature (sample retention + a new rule version + verifier branch +
tests mirroring `__tests__/evolve-proof.test.ts`), not a one-liner — and it changes a
replay-verified core path. It deserves an explicit decision before implementing.

## Recommendation

The honest outcome is: **no redundant PR.** If a contribution is still wanted, the
candidate-selection ANOVA veto is the genuinely-new option, but it's a meaningful change
to Ruv's replay-verified promotion system — scope it deliberately, don't rush it.
