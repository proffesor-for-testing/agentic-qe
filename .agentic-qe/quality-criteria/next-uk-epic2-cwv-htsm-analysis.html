Performance
"How speedy and responsive is it under real-world conditions?"

💡 Why This Matters

This epic exists primarily because of performance. The acceptance criteria define 4 explicit Core Web Vitals thresholds (LCP < 2.5s, FID < 100ms, INP < 200ms, CLS < 0.1) and a 40% page weight reduction target. Google uses CWV as a ranking signal — the current ~60% pass rate means approximately 40% of page loads fail Google's thresholds, directly suppressing search visibility for a site that competes with ASOS, M&S, and John Lewis for identical product keywords.

Business Impact: Improving CWV pass rate from 60% to 90%+ is projected to reduce homepage bounce rate from 45% to 35% (a 22% relative reduction). At Next's estimated ~15 million monthly UK homepage visits, a 10 percentage-point bounce rate reduction retains approximately 1.5 million additional sessions per month. Combined with the search conversion rate target increase from 2.5% to 4%, this translates to an estimated 60% uplift in search-driven revenue attributable to this epic alone.

Capability
"Can it perform the required functions — specifically, can users access content and navigate without JavaScript?"

💡 Why This Matters

The current Next.co.uk homepage shows "You need to enable JavaScript to run this app" — this means zero content is available to search engine crawlers relying on initial HTML, users with JavaScript disabled or blocked by corporate firewalls, assistive technologies that rely on static HTML, and any browser where JS fails to execute (network error, script error, CSP block). The primary deliverable of this epic is to make the homepage functionally useful without JavaScript. This is a binary pass/fail: either the SSR/SSG implementation renders product categories, search, and navigation in the initial HTML response, or it does not.

Business Impact: Google's renderer has a crawl budget; pages requiring JS execution consume approximately 5-10x more render resources and are indexed with significant delay (hours to weeks). For a catalogue with thousands of product pages linked from the homepage, this indexing delay directly suppresses organic search traffic. Estimated impact: 15-25% of organic search impressions are lost due to JS-only rendering, based on industry benchmarks for large e-commerce sites.

Epic AC #5: Above-the-fold content served as static HTML with critical CSS inlined Direct Requires both SSR HTML and critical CSS extraction Inlined critical CSS eliminates render-blocking stylesheet requests; but over-inlining bloats HTML response size and can itself delay first meaningful paint — requires careful extraction of only above-the-fold styles

⚠️ Feature flags are proposed for phased rollout, but the no-JS capability is binary: the page either renders server-side HTML or it does not. Feature flags can gate new SSR routes but cannot provide a partial SSR state — careful route-by-route migration strategy is needed.

Reliability
"Will it work well and resist failure during the SSR migration and under real-world conditions?"

💡 Why This Matters

Migrating from a client-side SPA to SSR fundamentally changes the failure surface. Server-side rendering introduces new failure modes: SSR server crashes, memory leaks during rendering, template compilation errors, hydration mismatches, and CDN cache staleness. The current SPA pushes all rendering to the client; with SSR, the server becomes a single point of failure for initial page delivery. If the SSR server fails, users get nothing — worse than the current SPA which at least serves a loading shell.

Business Impact: Next.co.uk processes an estimated 2-3 million daily sessions. A 15-minute SSR server outage during peak hours (7-9 PM UK time) would affect approximately 30,000-50,000 sessions. Without graceful degradation to the client-side SPA, this represents a complete loss of those sessions at an estimated average order value of 55 GBP, with potential revenue impact of 40,000-80,000 GBP per incident.

📊 Evidence Analysis

Source Reference Type Quality Implication Reasoning
Epic Risk: "SSR migration may break existing features" Direct Acknowledged risk of feature breakage during migration SSR hydration errors are the #1 cause of post-migration reliability issues; they manifest as broken event handlers, duplicated DOM nodes, or complete page re-renders that negate SSR performance gains
Risk Mitigation: "Feature flags, phased rollout" Direct Phased rollout reduces blast radius but requires robust rollback Feature flags mitigate risk only if rollback is tested and instant; if flag toggling requires redeployment or cache purge, the effective rollback time may exceed acceptable downtime thresholds
Technical Task #2: CDN with edge caching Inferred CDN cache staleness could serve outdated or broken content If SSR output is cached at CDN edge and the SSR server deploys a breaking change, stale cache entries may serve inconsistent content. Cache invalidation strategy must account for SSR version mismatches — requires verification of cache TTL and purge mechanism

⚠️ Risk Warnings

⚠️ No fallback strategy is documented for SSR server failure. If the SSR process crashes, the system must either serve a cached version, fall back to client-side rendering, or serve a meaningful static error page — not a blank page.
⚠️ SSR memory leaks are common in Node.js-based renderers (React renderToString, Vue renderToString). Without memory monitoring and automatic process recycling, SSR servers can degrade progressively and fail under sustained load.

📋 Recommended Actions

▼
Engineering
Implement SSR fallback chain: SSR → CDN cache → CSR → static shell

Configure 3-tier fallback: (1) live SSR, (2) stale CDN cache with stale-while-revalidate header, (3) client-side SPA bundle, (4) static HTML shell with core navigation. Each tier should log its activation for monitoring.

Testing
Chaos testing for SSR failure modes

Simulate SSR server crash, memory exhaustion, and slow API responses. Verify fallback chain activates correctly. Test hydration mismatch detection with intentional server/client divergence. Measure recovery time from each failure mode.

Engineering
Configure SSR process health monitoring and auto-recycling

Set memory threshold alerts at 80% and automatic process restart at 90%. Monitor SSR render time p95/p99 and alert if render time exceeds 500ms (indicating memory pressure or template complexity growth).

Security
"How well protected against unauthorized use, injection, and cache-based attacks?"

💡 Why This Matters

SSR introduces server-side attack surfaces that do not exist in a client-side SPA: server-side template injection (SSTI), where user input rendered on the server can execute arbitrary code; CDN cache poisoning, where a manipulated request causes the CDN to cache and serve malicious content to all users; and Content Security Policy (CSP) challenges from inlining critical CSS (AC #5), which requires unsafe-inline or nonce-based CSP headers. These are qualitatively different from client-side XSS because they affect all users, not just the attacking user.

Business Impact: A CDN cache poisoning attack on the Next.co.uk homepage would serve malicious content to all 15 million monthly visitors until the cache is purged. A server-side template injection could expose server environment variables, API keys, or customer data. UK GDPR fines for data breaches can reach 17.5 million GBP or 4% of annual turnover, whichever is higher.

📊 Evidence Analysis

Source Reference Type Quality Implication Reasoning
Epic AC #5: Critical CSS inlined Inferred Inlined CSS requires CSP policy adjustment Inlining CSS means the existing CSP (if any) must allow inline styles via 'unsafe-inline' (weak) or nonce/hash-based allowlisting (strong). If CSP is not updated, inline styles will be blocked, breaking above-the-fold rendering — requires verification of current CSP configuration
Technical Task #1: Implement SSR/SSG Inferred SSR introduces server-side template injection risk If any user-generated content (search queries, URL parameters) is rendered server-side without sanitization, it could enable SSTI or reflected XSS at the server level, affecting all subsequent cached responses
Technical Task #2: CDN with edge caching Inferred CDN cache poisoning risk if cache key includes manipulable request headers If the CDN varies cache by headers like X-Forwarded-Host or custom headers that an attacker can control, a poisoned response could be cached and served to all users. Cache key design must be explicitly reviewed — requires verification of CDN cache key configuration

⚠️ Risk Warnings

⚠️ The epic does not mention security considerations, CSP policy, or input sanitization requirements. Security is absent from acceptance criteria, technical tasks, and risk mitigation. This gap must be addressed explicitly.
⚠️ CDN cache poisoning is a particularly dangerous attack because it converts a single-user vulnerability into a broadcast attack affecting all visitors. Cache key audit and cache-control header review are mandatory before go-live.

📋 Recommended Actions

▼
Security
Implement nonce-based CSP for inlined critical CSS

Generate a per-request nonce for inline style tags. Update CSP header to allow style-src 'nonce-{value}' instead of 'unsafe-inline'. Verify CSP does not block SSR-rendered content. Test with CSP report-only mode first.

Security
Audit CDN cache key configuration for cache poisoning vectors

Review which request headers are included in CDN cache key (Host, Accept, Accept-Encoding are safe; X-Forwarded-*, custom headers are risky). Ensure cache key cannot be manipulated by untrusted request parameters. Implement Cache-Control: private for any user-specific content.

Security
Sanitize all server-rendered user input

Audit SSR templates for any injection of URL parameters, search queries, or user-provided data. Apply context-aware output encoding (HTML entity encoding for HTML context, CSS encoding for style context). Use framework-native auto-escaping and verify it is not disabled.

Testing
Penetration test SSR endpoints for SSTI and cache poisoning

Run targeted pentest before production launch covering: (1) template injection via URL params, (2) cache poisoning via header manipulation, (3) CSP bypass attempts, (4) response splitting via SSR headers. Engage security team or external assessor.

Scalability
"How well does it handle increased demand, especially SSR rendering at peak traffic?"

💡 Why This Matters

The current SPA architecture offloads all rendering to the client browser. SSR shifts rendering cost to the server — every page request now requires server CPU and memory to render HTML. During peak traffic events (Black Friday, sale launches, marketing campaigns), the SSR server fleet must handle the full request volume. UK retail peak traffic can reach 5-10x normal daily traffic, meaning SSR infrastructure must handle 10-20 million renders per day during peak periods.

Business Impact: Next.co.uk's peak trading days (Black Friday, Boxing Day sale) generate estimated 3-5x normal revenue. If SSR servers cannot scale to meet demand and the fallback chain fails, revenue loss during a 1-hour peak outage could reach 200,000-500,000 GBP based on Next plc's published online revenue figures.

📊 Evidence Analysis

Source Reference Type Quality Implication Reasoning
Technical Task #1: Implement SSR/SSG Inferred SSR moves rendering cost from client to server, creating new scaling requirements SSG (static generation) eliminates runtime rendering cost entirely; SSR requires per-request rendering. The choice between SSR and SSG determines the scaling profile — SSG scales infinitely via CDN, SSR requires horizontal server scaling
Technical Task #2: CDN with edge caching Direct CDN edge caching absorbs traffic spikes if cache hit rate is high With proper cache-control headers and long TTLs, the CDN can serve 95%+ of homepage requests from cache, reducing SSR server load by 20x. But cache invalidation during deployments creates a thundering herd problem when all CDN nodes simultaneously re-request from origin

⚠️ Risk Warnings

⚠️ Cache invalidation during deployment creates thundering herd: if CDN cache is purged globally during a release, all edge nodes simultaneously request fresh SSR content from origin, potentially overwhelming the SSR fleet.

📋 Recommended Actions

▼
Engineering
Favor SSG over SSR for homepage where possible

If homepage content changes less than hourly, use Incremental Static Regeneration (ISR) or build-time SSG with revalidation intervals. This eliminates runtime SSR cost and makes the homepage infinitely scalable via CDN.

Performance
Load test SSR at 5x peak traffic with cache miss simulation

Simulate Black Friday traffic (5x normal) with CDN cache disabled to stress-test SSR server fleet. Measure render time p95/p99, error rate, and memory consumption. Define auto-scaling triggers based on results.

Engineering
Implement staggered cache invalidation

Instead of global cache purge on deployment, use soft purge (stale-while-revalidate) or rolling invalidation across CDN PoPs over 5-10 minutes to prevent thundering herd on origin.
